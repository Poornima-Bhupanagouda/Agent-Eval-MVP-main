## 2. REPORT DATA MODEL

### 2.1 ReportData
```python
@dataclass
class ReportData:
    title: str              # Report title (suite name, batch name, or "Quick Test")
    endpoint: str           # Agent endpoint tested
    total_tests: int        # Count of tests in report
    passed_tests: int       # Count passing threshold
    failed_tests: int       # Count failing threshold
    avg_score: float        # Mean score across all tests
    avg_latency_ms: float   # Mean agent response time
    results: List[Dict]     # Per-test result details
    generated_at: str       # ISO timestamp (auto-set)
```

### 2.2 Per-Result Dict Structure
Each item in `results[]`:
```python
{
    "input": str,           # Test input sent to agent
    "output": str,          # Agent response
    "expected": str,        # Expected answer (optional)
    "score": float,         # Overall score (0-100)
    "passed": bool,         # Pass/fail status
    "latency_ms": int,      # Response time
    "evaluations": [        # Per-metric breakdown
        {"metric": str, "score": float, "passed": bool, "reason": str}
    ]
}
```

## 3. HTML REPORT DESIGN

### 3.1 Requirements
* Self-contained ‚Äî all CSS inline, no external dependencies
* Lilly-branded ‚Äî red accent (#D52B1E), white background, professional typography
* Printable ‚Äî proper print CSS for PDF generation via browser
* Responsive ‚Äî readable on desktop and tablet

### 3.2 Report Sections
1. **Header** ‚Äî Report title, generation timestamp, Lilly logo placeholder
2. **Executive Summary** ‚Äî Total tests, pass rate, avg score, avg latency (4-card grid)
3. **Pass Rate Gauge** ‚Äî Visual indicator (green/yellow/red)
4. **Test Results Table** ‚Äî Per-test: status icon, input, output, score, latency
5. **Evaluation Details** ‚Äî Per-metric chips with score and pass/fail color
6. **Footer** ‚Äî "Generated by Lilly Agent Eval" + timestamp

### 3.3 Color Coding
| Score Range | Color | CSS Class |
|-------------|-------|-----------|
| 90-100 | Green (#22c55e) | `.pass` |
| 70-89 | Yellow (#f59e0b) | `.warning` |
| 0-69 | Red (#ef4444) | `.fail` |

### 3.4 HTML Escaping
* All user content (input, output, expected) escaped via `_escape_html()`
* Never use f-string interpolation with unescaped user data in HTML
* `_escape_html()` replaces: `&`, `<`, `>`, `"`, `'`

## 5. API ENDPOINTS

### 5.1 HTML Report
* `GET /api/reports/html?suite_id=X` or `?batch_id=X`
* Returns `HTMLResponse` with Content-Disposition for download
* Filename: `eval_report_{timestamp}.html`

### 5.2 JSON Report
* `GET /api/reports/json?suite_id=X` or `?batch_id=X`
* Returns `JSONResponse` with report data
* Used for programmatic access and CI/CD integration

### 5.3 CSV Export (History)
* `GET /api/history/export?format=csv`
* Returns CSV with columns: timestamp, input, output, score, passed, latency, endpoint
* Uses streaming response for large result sets

## 7. WHAT TO AVOID

* External CSS/JS dependencies ‚Äî reports must work offline
* Unescaped user content in HTML ‚Äî XSS risk even in downloaded reports
* Large inline images ‚Äî keep report file size reasonable
* Framework templating (Jinja) for reports ‚Äî use Python f-strings for self-contained output
* Non-UTF-8 encoding ‚Äî always encode as UTF-8

---

## 8. TESTING REPORT GENERATION

* Empty results ‚Üí should generate valid HTML with "No results" message
* Single result ‚Üí full report with one test card
* 100+ results ‚Üí report should render without performance issues
* Special characters in input/output ‚Üí must be HTML-escaped
* Missing evaluations ‚Üí should show "No evaluations" gracefully
* Open generated HTML in browser to visually verify layout
